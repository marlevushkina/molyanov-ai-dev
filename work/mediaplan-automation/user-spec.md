# User Spec: Автозаполнение медиаплана из GetSales

## Что делаем
Google Apps Script, который ежедневно забирает статистику LinkedIn-кампаний из GetSales API и записывает метрики воронки в Google Sheets медиаплана. Поддержка нескольких клиентов с одним аккаунтом GetSales.

## Зачем
Сейчас метрики воронки (invites sent, connects, message replies) заполняются вручную каждый день. Это занимает время и приводит к ошибкам. Автоматизация освобождает время для анализа результатов вместо рутинного сбора данных.

## Как должно работать
1. Каждый день в 08:00 (Dubai time) скрипт запускается автоматически
2. Скрипт получает список кампаний из GetSales API
3. Для каждого клиента (EasyStaff, Core 24/7) открывает соответствующую Google-таблицу
4. Читает строки с гипотезами, пропуская статусы Draft и Stop
5. Матчит гипотезу с кампанией в GetSales по ID (UAE-01, KZ-SMB-CTO-01)
6. Получает статистику по нодам воронки (invite → messages)
7. Записывает: invites sent, connects, msg1/2/3 sent & replies, total replies
8. Рассчитывает CR-метрики: CR connects, reply rate, positive rate, lead rate
9. При достижении 100 и 200 инвайтов — фиксирует checkpoint (дата, connects, replies)
10. Ставит дату последнего обновления

## Критерии приёмки
- [ ] Скрипт корректно подключается к GetSales API и получает список кампаний
- [ ] Маппинг гипотез → кампании работает по ID (короткий UAE-01 и длинный KZ-SMB-CTO-01)
- [ ] Метрики записываются в правильные колонки (формат ES: 51 колонка, A-AY)
- [ ] Строки со статусом Draft и Stop пропускаются
- [ ] Checkpoints на 100 и 200 инвайтов фиксируются однократно
- [ ] CR-метрики рассчитываются корректно
- [ ] Ежедневный триггер работает в 08:00 Asia/Dubai
- [ ] Поддержка нескольких клиентов (разные spreadsheet + sheet)
- [ ] Ручные данные (positive, leads, MQL) не перезаписываются

## Ограничения
- Один аккаунт GetSales (smmekalka@gmail.com), один API-ключ на все кампании
- Таблицы принадлежат klevrs.project@gmail.com
- Google Apps Script: лимит 6 мин на выполнение, лимит на UrlFetch
- GetSales API: нет webhook-ов, только polling
- Кампании в GetSales должны быть названы по ID гипотезы

## Риски
- **Риск 1:** Кампании в GetSales названы в старом формате (ES_UAE_T_1 вместо UAE-01). **Митигация:** Пользователь переименовывает кампании; скрипт поддерживает оба формата через regex.
- **Риск 2:** Apps Script не открывается из аккаунта klevrs.project. **Митигация:** Создан новый скрипт-проект привязанный к правильному аккаунту; альтернатива — standalone script.
- **Риск 3:** API GetSales может измениться. **Митигация:** Отладочные функции (debugGetSalesResponse, debugFlowMapping) позволяют быстро диагностировать проблемы.

## Технические решения
- Мы решили использовать Google Apps Script, потому что он нативно работает с Google Sheets и не требует отдельного хостинга.
- Мы решили использовать один API-ключ на всех клиентов, потому что все кампании в одном аккаунте GetSales.
- Мы решили унифицировать формат таблиц Core под формат ES (51 колонка), потому что это упрощает скрипт — один маппинг колонок на всех.
- Мы решили не использовать формулы в ячейках для CR-метрик, а считать их в скрипте, потому что формулы сбиваются при ручном редактировании.

## Тестирование

**Unit-тесты:** Не применимо (Google Apps Script не имеет встроенного тест-фреймворка).

**Интеграционные тесты:** debugGetSalesResponse() — проверяет подключение к API и парсинг ответа.

**E2E тесты:** debugFlowMapping() — проверяет полный маппинг гипотез из таблиц к кампаниям GetSales.

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. Запустить debugGetSalesResponse | Apps Script Editor | Лог показывает список кампаний и статистику |
| 2. Запустить debugFlowMapping | Apps Script Editor | Каждая гипотеза сматчена с кампанией или показано НЕ НАЙДЕНА |
| 3. Запустить syncGetSalesToSheet | Apps Script Editor | Метрики записаны в таблицу, лог показывает кол-во обновлённых строк |

### Пользователь проверяет
- Открыть таблицу EasyStaff → лист "Гипотезы" → проверить, что колонки Z-AK заполнены актуальными данными
- Открыть таблицу Core → лист "Гипотезы (unified)" → аналогично
- Проверить, что ручные данные (positive, leads) не затёрты
